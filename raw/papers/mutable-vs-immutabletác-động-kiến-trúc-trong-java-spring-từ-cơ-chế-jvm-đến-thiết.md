Mutable vs. ImmutableTác động Kiến trúc trong Java & Spring Từ Cơ chế JVM đến Thiết kế Hệ thống Cấp cao 1. Định nghĩa Cốt lõi & Cơ chế JLS MUTABLE (KHẢ BIẾN): Trạng thái thay đổi sau khi khởi tạo. Sửa đổi tại chỗ (In-place modification) trong vùng nhớ Heap.Ví dụ: StringBuilder sb = new StringBuilder("Hello"); sb.append(" World"); (Sử dụng cùng một tham chiếu).IMMUTABLE (BẤT BIẾN): Trạng thái cố định ngay khi hàm khởi tạo hoàn tất. Thay đổi giá trị đồng nghĩa với việc tạo đối tượng mới.Ví dụ: String s = "Hello"; s = s.concat(" World"); (Tạo tham chiếu mới).Technical Deep Dive: Trong Java Memory Model, cơ chế trường final đảm bảo 'freeze action', giúp các luồng (threads) khác nhìn thấy giá trị khởi tạo an toàn mà không cần đồng bộ hóa.2. Chi phí Ẩn của Tính Khả biến (Concurrency) Mutable State + Multi-threading = Race Conditions Việc sử dụng Synchronized Lock (Mutable Object) gây ra tình trạng "Traffic Jam" (kẹt xe) và lãng phí CPU (CPU Waste) khi nhiều luồng cùng truy cập.Các vấn đề bao gồm:Contention (Tranh chấp): Các luồng bị chặn chờ tài nguyên.Context Switching: Chi phí Hệ điều hành (OS) phải chuyển đổi giữa các luồng.CPU Cache Invalidation: Rào cản bộ nhớ buộc CPU phải xóa cache $L1/L2$.Ngược lại, Immutable Object mang lại trạng thái Zero Contention (Không tranh chấp), cho phép các luồng truy cập song song mà không bị chặn.3. Tác động Bộ nhớ & Myth về Hiệu năng The Myth: "Tạo đối tượng mới là chậm." The Reality: Phân bổ trên TLAB (Thread Local Allocation Buffer) gần như miễn phí (pointer bump). Modern GC (Garbage Collector) được tối ưu hóa cho các đối tượng bất biến ngắn hạn.Cơ chế GC: _ Thu gom rác cực nhanh (G1/ZGC) cho đối tượng chết trẻ ở vùng Young Gen.Đối tượng sống lâu ở vùng Old Gen gây phân mảnh bộ nhớ.String Pool Optimization: Tính bất biến cho phép áp dụng Cache/Flyweight pattern, giúp tiết kiệm bộ nhớ Heap đáng kể.4. public class UserService { [cite: 58]

Disaster Timeline (Hậu quả): Nếu Request A gán currentUser = 'Alice' và Request B gán currentUser = 'Bob', khi Request A đọc lại giá trị sẽ thấy 'Bob' $\rightarrow$ Gây ra hiện tượng Ghost Data (Dữ liệu Ma).Quy tắc: Singleton phải STATELESS. Trạng thái phải nằm trong Stack (biến cục bộ) hoặc DTO.5. Mô hình hóa Dữ liệu: Từ Lombok đến Java Records Lombok @Data (Không khuyến khích cho DTO):Tạo ra Mutable Object thông qua Setters.Sinh ra nhiều boilerplate code (Heavy boilerplate generated).Rủi ro: Dữ liệu có thể bị sửa đổi ngẫu nhiên.Java Records (Java 16+) (Khuyên dùng):Javapublic record UserDto (String name, int age) {} [cite: 89]
Mặc định là Immutable (Immutable by default).Hoạt động như những Transparent carriers với cú pháp ngắn gọn.Benchmark: Việc sử dụng Jackson Serialization với Records nhanh hơn từ 12-15% do sử dụng canonical constructor.6. Quản lý Cấu hình Bất biến The Mutable/Bad Way: Sử dụng @ConfigurationProperties (POJO) với Setter Injection. Khuyết điểm là đoạn code khác có thể gọi setConfig(...) làm thay đổi cấu hình khi runtime.The Immutable/Good Way: Sử dụng @ConstructorBinding / Record với Constructor Injection. Cách này đảm bảo an toàn và tính nhất quán cho môi trường.Java@ConfigurationProperties("app") [cite: 105]
public record AppConfig(String url, int timeout) {} [cite: 106] 7. Chiến lược Dependency Injection Field Injection (Tránh sử dụng):Java@Autowired [cite: 110]
private Service service; [cite: 111]
Nhược điểm: Mutable, tiềm ẩn rủi ro NullPointerException, và che giấu vòng lặp phụ thuộc.Constructor Injection (Khuyên dùng):Javaprivate final Service service; [cite: 113]

Ưu điểm: Immutable (nhờ từ khóa final), Atomic Initialization, và dễ dàng để test.8. An toàn luồng trong Collections Visual Metaphor: Bức tường phòng thủ (The Defensive Wall).Scenario A (Unsafe): return this.items; $\rightarrow$ Trả về trực tiếp tham chiếu danh sách thực tế bên trong object, làm lộ trạng thái.Scenario B (Defensive Copy): return List.copyOf(this.items); $\rightarrow$ Trả về một Unmodifiable List an toàn.Quy tắc: Không bao giờ tin tưởng Collection được truyền vào hoặc trả ra. Luôn sử dụng Defensive Copying.9. Tính Nguyên tử trong Map: ConcurrentHashMap Lỗi "Check-then-Act": Nếu sử dụng if (!map.containsKey(key)) { map.put(key, val); }, sẽ xảy ra Race Condition khi 2 thread cùng kiểm tra và cùng ghi đè lên nhau.Giải pháp (Atomic Operation): Sử dụng map.computeIfAbsent(key, k -> factory(k));.So sánh các loại Map:HashMap: Không an toàn luồng (Not Thread-safe).Hashtable: Chậm do sử dụng Global Lock.ConcurrentHashMap: Nhanh nhờ cơ chế Segmented Locking (Khuyên dùng).10. Sao chép Sâu vs. Sao chép Nông (Deep vs. Shallow Copy) Sao chép Nông (Shallow Copy): Cả bản gốc (Parent A) và bản sao (Copy B) đều chia sẻ chung một trạng thái (Shared Mutable State). Nếu phần tử con (Child) thay đổi, cả hai sẽ bị ảnh hưởng.Sao chép Sâu (Deep Copy): Đảm bảo tính độc lập hoàn toàn (Total independence) giữa bản gốc và bản sao.Phương pháp Copy:Copy Constructors: Khuyên dùng nhất (Best).Serialization: Chậm (Slow).Cloning: Bị hỏng, nên tránh (Broken - Avoid).11. Truyền Ngữ cảnh: ThreadLocal vs. ScopedValue ThreadLocal (Legacy): Mutable, tiềm ẩn nguy cơ rò rỉ bộ nhớ (Memory Leaks) với Thread Pooling, và chi phí cao khi kết hợp với Virtual Threads.ScopedValue (Java 21 / Modern): _ Cú pháp: ScopedValue.where(KEY, value).run(() -> { ... }).Đặc tính: Immutable, vòng đời bị giới hạn (Bounded Lifetime), gọn nhẹ và tương thích tốt với Virtual Thread (Lightweight).12. Ma trận Quyết định (Decision Matrix) Use IMMUTABLE For: Use MUTABLE For: - DTOs / Value Objects (Records) - System Configuration - Map Keys - Caching / Multi-threaded Access - JPA Entities (Bắt buộc bởi Hibernate) - Local Method Scope (Tối ưu hiệu năng vòng lặp) - Builders / Buffers (Giai đoạn khởi tạo/Construction phase) 13. Quy tắc Ngón tay cái (Rules of Thumb) Default to Immutable: Sử dụng Java Records cho mọi DTO.Singleton Purity: Các Spring Beans phải tuyệt đối Stateless.Final Fields: Luôn ưu tiên sử dụng Constructor Injection.Modern Context: Ưu tiên ScopedValue hơn ThreadLocal trong các ứng dụng Java 21+.Defensive Copy: Luôn bảo vệ các collection ở ranh giới của public API.
